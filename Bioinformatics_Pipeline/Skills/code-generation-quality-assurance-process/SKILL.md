# Policy: Code Generation Quality Assurance

**Version:** 2.1
**ID:** AROS-POLICY-CG-QA-V2.1

---

## 1. Preamble

This document establishes the mandatory quality assurance (QA) policy for all executable code generated by agents within the Antigravity Research OS (AROS). Adherence to this policy is critical for ensuring the reliability, security, maintainability, and correctness of all automated code production.

This policy codifies a strict, multi-stage verification pipeline to prevent the introduction of flawed, insecure, or incomplete code into the AROS ecosystem.

## 2. GEPA Error Prevention Rule: Multi-Stage Verification

**Rule ID:** GEPA-Rule-CG-QA-002

**Statement:** All code generation tasks MUST implement a strict multi-stage verification pipeline (Generation -> QA -> Review). Code is not considered complete, correct, or ready for deployment until it has successfully passed all three stages. Bypassing any stage is a critical policy violation.

## 3. The Multi-Stage Verification Pipeline

The following three-stage process is mandatory for any agent or workflow that generates executable code, scripts, or configuration files.

### 3.1. Stage 1: Generation & Self-Correction

This initial stage focuses on creating functionally correct code and ensuring its intrinsic quality through a tight feedback loop.

1.  **Code Generation:** The agent generates the initial code based on the user's request, source data, and relevant skills.
2.  **Test Generation:** The agent MUST write a corresponding suite of unit tests that validate the core functionality of the generated code.
3.  **Static Analysis:** The agent MUST perform static analysis (e.g., using `pylint`, `flake8`, `shellcheck`) to ensure the code adheres to AROS coding standards and is free of common syntactical and stylistic errors.
4.  **Self-Correction Loop:** The agent MUST execute the tests and static analysis. If any checks fail, the agent MUST iteratively revise the code and re-run all checks until they pass successfully.

### 3.2. Stage 2: Automated QA Gate (GTB)

Once the code is internally consistent, it must be validated against a broader set of system-wide quality and safety standards. This stage is a **non-negotiable** automated gate.

1.  **Submission:** The generated code, along with its tests, MUST be submitted to the Golden Test Battery (GTB) using the `iterative-validator` skill, which orchestrates the validation process.
2.  **Task Type:** The `task_type` parameter for the validator MUST be set to `"code_generation"`.
3.  **Evaluation:** The `iterative-validator` handles the pass/fail logic internally. A successful exit from the skill indicates the code has passed this stage.

### 3.3. Stage 3: Review

This stage ensures that the code not only works but is also well-designed, maintainable, and aligned with the project's goals.

1.  **Peer Agent Review:** The code MUST undergo a review by a second, independent agent. This process is now programmatically invoked and managed by the `iterative-validator` skill's external feedback mechanism. If the automated QA gate (Stage 2) requires more than two self-correction attempts, the `iterative-validator` automatically solicits this peer review to provide new feedback for a final remediation attempt. This connects the QA process directly to the review stage, ensuring that difficult-to-fix code gets a second opinion.

## 4. GEPA Error Prevention Rule: Standardized Deployment Architecture

**Rule ID:** GEPA-Rule-CG-QA-003

**Statement:** Adhere to a standard server architecture (e.g., MCP) for all agent package deployments. This ensures consistency, prevents configuration drift, and simplifies deployment, reducing deployment-related errors.
