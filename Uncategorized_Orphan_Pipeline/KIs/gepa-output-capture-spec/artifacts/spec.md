# GEPA Policy: Output Capture and Availability - Technical Specification

## 1. Overview

This document provides the precise technical specifications for implementing the `OutputCaptureAndAvailability` policy, a component of the General Environment Pre-flight Assessment (GEPA) framework. The objective is to ensure that all outputs generated by an AROS agent during a task are reliably captured, structured, and made available for subsequent evaluation, debugging, and archival processes.

This specification addresses three core areas:
1.  **Output Hooking**: The mechanisms used to intercept all agent-generated data, including standard streams and file system artifacts.
2.  **Logging Format**: A standardized, machine-readable format for recording all captured output events.
3.  **Exposure Mechanism**: The defined methods for making these logs accessible to other AROS system components.

## 2. Output Capture Mechanisms

To ensure comprehensive capture, the AROS execution environment will impose the following standards on all agent-run tasks.

### 2.1. Dedicated Task Directory

For each assigned task or job, the AROS orchestrator MUST create a unique directory at the following path: `~/.gemini/antigravity/logs/jobs/<job_id>/`.

The agent MUST be launched with the environment variable `AROS_OUTPUT_DIR` set to this path. All file-based outputs (e.g., generated code, data files, reports) MUST be written exclusively within this directory. Direct writes to other locations are a policy violation.

### 2.2. Standard Stream Redirection

The process that invokes the agent is responsible for redirecting the agent's `stdout` and `stderr` streams.
- Each line emitted to `stdout` MUST be captured as a distinct log event.
- Each line emitted to `stderr` MUST be captured as a distinct log event.
- These streams must not be allowed to print directly to the console of the parent process.

### 2.3. Tool and Function Call Interception

The agent's execution harness MUST log the inputs and outputs of every tool call (e.g., `run_shell_command`, `write_local_file`). This ensures that even actions that don't produce `stdout` or files are recorded.

## 3. Structured Log Format

All captured output events for a given job MUST be written to a single, append-only log file named `output_capture.log.json` located inside the `AROS_OUTPUT_DIR`.

Each entry in this file is a discrete JSON object, written on a new line (JSONL format).

### 3.1. Log Entry Schema

| Field         | Type     | Description                                                                                             | Required |
|---------------|----------|---------------------------------------------------------------------------------------------------------|----------|
| `timestamp`   | String   | ISO 8601 timestamp of the event (e.g., `2023-10-27T10:00:00.123Z`).                                       | Yes      |
| `job_id`      | String   | The unique identifier for the overarching task.                                                         | Yes      |
| `agent_id`    | String   | Identifier for the specific agent instance handling the task.                                           | Yes      |
| `event_type`  | String   | The type of event being logged. See `Event Types` section below.                                        | Yes      |
| `payload`     | Object   | The data associated with the event. The structure of this object depends on the `event_type`.           | Yes      |
| `metadata`    | Object   | Optional field for additional context (e.g., process ID, execution step).                               | No       |

### 3.2. Event Types and Payload Schemas

#### `event_type: "stdout"` or `"stderr"`
```json
{
  "payload": {
    "line": "Text content of the output line."
  }
}
```

#### `event_type: "file_write"`
This event is logged when a file is created or modified within the `AROS_OUTPUT_DIR`.
```json
{
  "payload": {
    "filepath": "path/relative/to/AROS_OUTPUT_DIR/file.txt",
    "size_bytes": 1024,
    "sha256_hash": "a1b2c3d4..."
  }
}
```

#### `event_type: "tool_call"`
```json
{
  "payload": {
    "tool_name": "run_shell_command",
    "parameters": {
      "command": "ls -l"
    }
  }
}
```

#### `event_type: "tool_response"`
```json
{
  "payload": {
    "tool_name": "run_shell_command",
    "result": {
      "stdout": "total 0...",
      "stderr": "",
      "exit_code": 0
    }
  }
}
```

#### `event_type: "final_output"`
The agent's final, conclusive output for the entire task.
```json
{
  "payload": {
    "status": "SUCCESS",
    "message": "Task completed. Generated report and data files.",
    "artifacts": [
      "report.pdf",
      "data.csv"
    ]
  }
}
```

## 4. Log Availability and Access

To facilitate evaluation and system monitoring, the captured logs will be exposed through two primary mechanisms.

### 4.1. Direct Filesystem Access

Evaluation agents and other system processes can directly access the log file at its canonical location:
`~/.gemini/antigravity/logs/jobs/<job_id>/output_capture.log.json`

This method is suitable for automated, script-based processing.

### 4.2. AROS Dashboard API Endpoint

A new endpoint will be added to the AROS Dashboard Control service to provide programmatic access to the logs over HTTP. This aligns with existing system architecture and allows for remote access.

**Endpoint:** `GET /api/jobs/<job_id>/output`

**Success Response (200 OK):**
```json
{
  "job_id": "...",
  "log_file_path": "/home/user/.gemini/antigravity/logs/jobs/.../output_capture.log.json",
  "log_content": [
    { "timestamp": "...", "event_type": "stdout", ... },
    { "timestamp": "...", "event_type": "file_write", ... }
  ]
}
```

**Failure Response (404 Not Found):**
If the `job_id` does not exist.
```json
{
  "error": "Job ID not found",
  "job_id": "..."
}
```
This specification is now active. All agent development and execution environments must be updated to comply with these requirements.
