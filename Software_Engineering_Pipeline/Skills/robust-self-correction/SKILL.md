# Policy: Robust Self-Correction

## 1. Preamble

This policy establishes the fundamental requirement for all AROS agents to operate with resilience and autonomy. The primary directive is to achieve task objectives, and this often requires navigating unforeseen errors, API changes, or environmental inconsistencies. Agents are expected to handle these challenges without halting execution or requiring external intervention, within the bounds of this policy.

This policy directly supports the **GEPA Principle of "Strive for efficient task completion by ensuring all sub-tasks succeed on the first attempt, minimizing retries."** by providing a structured, auditable mechanism for overcoming initial failures.

## 2. GEPA Rule: Autonomous Error Correction & Reporting

This policy is governed by the following non-negotiable GEPA rule, which must be embedded in the agent's core operational logic.

> **The agent MUST identify and correct errors in its execution path or output without explicit external intervention. If self-correction fails after a defined number of attempts, the agent MUST escalate the failure through the appropriate diagnostic channels. The task is only considered failed after both self-correction and escalation have been completed.**

## 3. The Self-Correction Loop

When an agent encounters an error or deviation from the expected outcome, it MUST NOT terminate its task. Instead, it MUST initiate a **Self-Correction Loop**. This loop is a mandatory, iterative process of identifying, analyzing, and resolving issues to get the task back on track. The loop MUST NOT exceed a maximum of **3 attempts** (1 initial attempt + 2 retries) for a given error.

### 3.1. Triggers for the Self-Correction Loop

- **Tool Execution Failure:** A command returns a non-zero exit code or an API call fails.
- **Validation Failure:** Output generated by the agent fails a check (e.g., `gtb-validator`, linter, unit test), as mandated by the `code-generation-quality-assurance-process` policy.
- **Unexpected State:** The state of the environment does not match the expected state after an action.
- **Incomplete or Missing Data:** A required file, database entry, or piece of information is not found, and the `missing-input-handling-and-improvisation` policy cannot be safely applied.

### 3.2. The Correction Process

1.  **Error Identification & Logging:** Precisely capture the error message, exit code, and context of the failure. Log this information immediately to `aros.log` with a `[Self-Correction Attempt]` tag, including the attempt number (e.g., `[1/3]`).
2.  **Root Cause Analysis:** Go beyond the surface-level error. Use introspection, logging, and environmental checks to determine the underlying cause. (e.g., Is it a transient network issue? A permissions problem? A syntax error in generated code? A faulty assumption in the plan?).
3.  **Strategy Formulation:** Develop a concrete, actionable plan to resolve the issue. This may include:
    *   Re-running a command with corrected parameters.
    *   Querying `brain.db` or reading documentation to find a correct approach.
    *   Re-generating code or content based on the error feedback.
    *   Using an alternative tool or skill if one is available.
4.  **Execute & Verify:** Execute the corrective action. After execution, the agent MUST verify that the error has been resolved before proceeding with the original task.

## 4. Mandatory Escalation Protocol

If the Self-Correction Loop fails to resolve the error after the maximum number of attempts, the agent MUST trigger the **Escalation Protocol**. This is not a task failure but a state transition.

The protocol includes the following steps:
1.  **Log Critical Failure:** Log the unrecoverable error to `aros.log` with a `[Correction-Failed-Escalation]` tag, summarizing all attempted correction strategies.
2.  **State Persistence:** Save the current state of the task, including all generated artifacts and logs, to a designated recovery directory.
3.  **Invoke Fallback:** The agent must trigger a fallback mechanism as defined by the system architecture. This may involve:
    *   Handing off the task to a more capable 'specialist' agent or a larger model persona for advanced problem-solving.
    *   Creating a high-priority job in the `swarm_jobs` table in `brain.db` for asynchronous review.
    *   Halting the current task with a specific "escalated" status code to signal to the orchestrator that manual or higher-level intervention is required.

## 5. Policy Integration

This policy is a foundational component of the AROS operational logic and works in conjunction with other key policies:

-   **`code-generation-quality-assurance-process`**: Failures identified during the QA process (e.g., linting, testing, `gtb-validator`) are primary triggers for this self-correction policy.
-   **`iterative-validator`**: This skill is a key tool used within the self-correction loop for validating content quality. This policy governs the retry logic *around* the validator.
-   **`gepa_error_prevention_patch`**: This policy provides the high-level framework that the patch implements, defining the "why" behind the retry and fallback mechanisms.
